# Deployment

This repo is designed to run as a long-lived web service after setup. The service starts from the Dockerfile, exposes `GET /health`, receives events at `POST /webhook`, and runs the dispatcher against data-driven routes.

Once GitHub auto-deploy is connected, changes made by Claude Code, Codex, or a human to operations, prompts, or dispatcher code can be pushed live without a user managing servers.

## Recommended Platforms

Railway and Render are the recommended starting points for simple projects because they hide most infrastructure work. The engine is still just a Dockerized HTTP service, so teams can also run it on Kubernetes, Azure, GCP, AWS, or on-prem VMs when that better matches their security, procurement, or operating model.

| Rank | Platform | Recommendation | Notes |
|---:|---|---|---|
| 1 | Railway | Primary one-click path | Best fit for low-ops users and agent-assisted workflows. Uses `railway.toml`, the repo Dockerfile, GitHub auto-deploys, and `/health`. |
| 2 | Render | Best fallback | Simple dashboard and Blueprint flow. Uses `render.yaml`, the repo Dockerfile, GitHub auto-deploys, and `/health`. |
| 3 | DigitalOcean App Platform | Good conventional PaaS | Works with Dockerfile/GitHub deploys, but needs separate app spec setup if you want a first-class one-click path. |
| 4 | Northflank | Strong but more platform-like | Works well for teams that want services, jobs, databases, and environments in one control plane. More setup than Railway or Render. |
| 5 | Fly.io | Good for technical operators | Docker-native and production-capable, but usually requires `flyctl`, `fly.toml`, regions, and more CLI work. |
| 6 | AWS App Runner / ECS / Marketplace | Enterprise packaging path | Useful for procurement and AWS-native customers, but too much IAM/account/region surface for the default one-click flow. |

## Other Valid Targets

These are good choices when the buyer already has a preferred infrastructure platform. They are not the default one-click path for this repo, but the Dockerfile and `PORT` fallback make them straightforward.

| Target | Good for | What to run |
|---|---|---|
| Kubernetes | Platform teams, multi-tenant installs, private networking, enterprise controls | Build the Docker image, deploy it as a `Deployment`, expose `/webhook` through an ingress, and use `/health` for readiness/liveness checks. |
| Azure | Microsoft-first teams, Azure procurement, existing AKS/App Service/Container Apps estates | Use Azure Container Apps or App Service for simpler projects; use AKS when the team already operates Kubernetes. |
| GCP | Google Cloud teams, Cloud Run simplicity, GKE estates | Use Cloud Run for simple container hosting; use GKE when Kubernetes is already standard. |
| On-prem VM | Air-gapped, private network, or strict data residency deployments | Run the Docker image behind a reverse proxy with TLS, set env vars as service secrets, and keep `/webhook` reachable by Personize or the private gateway. |

Minimum requirements are the same everywhere: a public or private URL for `/webhook`, persistent environment secrets, outbound HTTPS to the selected Personize backend, and restart/health-check supervision.

## Railway

Railway is the preferred default for this repo because it is the shortest path from GitHub repo to running Docker service.

1. Fork or import this repo into GitHub.
2. Create a Railway project from the GitHub repo.
3. Railway reads `railway.toml` and builds with the Dockerfile.
4. Add required variables:

```bash
PERSONIZE_MODE=hosted
PERSONIZE_SECRET_KEY=sk_live_...
PERSONIZE_CRM_CONNECTION_ID=
PERSONIZE_WEBHOOK_SECRET=whsec_...
DRY_RUN=true
LOG_LEVEL=info
```

5. Deploy the service and copy the public domain.
6. Set:

```bash
PERSONIZE_WEBHOOK_URL=https://your-railway-domain/webhook
```

7. Register webhooks:

```bash
npm run setup:webhooks
```

When you are ready for live writes, change `DRY_RUN=false`.

## Render

Render is the recommended fallback because it has a clean Blueprint flow and a simple dashboard for nontechnical users.

1. Fork or import this repo into GitHub.
2. Create a new Render Blueprint from the repo.
3. Render reads `render.yaml` and builds with the Dockerfile.
4. Fill in the synced secrets:

```bash
PERSONIZE_SECRET_KEY=sk_live_...
PERSONIZE_CRM_CONNECTION_ID=
PERSONIZE_WEBHOOK_URL=https://your-render-domain/webhook
```

`PERSONIZE_WEBHOOK_SECRET` is generated by the Blueprint. Keep `DRY_RUN=true` until you have reviewed dry-run output.

After the service has its public URL, run:

```bash
npm run setup:webhooks
```

## Required Environment

| Variable | Required | Notes |
|---|---:|---|
| `PERSONIZE_MODE` | yes | `hosted` for Personize Cloud; `private` for Personize Private. |
| `PERSONIZE_SECRET_KEY` | hosted only | Personize Cloud API key. |
| `PERSONIZE_GATEWAY_URL` | private only | Personize Private gateway URL. |
| `PERSONIZE_GATEWAY_KEY` | private only | Bearer key for the private gateway. |
| `PERSONIZE_CRM_CONNECTION_ID` | optional | Pin a CRM connection when the org has more than one. |
| `PERSONIZE_WEBHOOK_SECRET` | yes | HMAC secret used to validate inbound webhooks. |
| `PERSONIZE_WEBHOOK_URL` | post-deploy | Public service URL with `/webhook`; needed by `setup:webhooks`. |
| `DRY_RUN` | recommended | Defaults safe. Set `false` only when ready for live writes. |
| `ENGINE_PORT` / `PORT` | platform | Local runs use `ENGINE_PORT`; managed hosts usually inject `PORT`. |

## Deployment Flow

The full rollout sequence is:

```bash
npm run setup:verify
npm run setup:diff -- --crm hubspot
npm run setup -- --crm hubspot
# deploy on Railway or Render
# set PERSONIZE_WEBHOOK_URL to https://your-service/webhook
npm run setup:webhooks
```

After that, two independent lanes run off the same `git push`:

- **Code** (`src/**`) — a push to the connected branch redeploys the engine via Railway/Render's own GitHub auto-deploy.
- **Content** (`manifests/**`: guidelines, collections, entity types, graph relations, and dispatch routes) — a push touching `manifests/**` triggers [`.github/workflows/publish-content.yml`](../.github/workflows/publish-content.yml), which runs `setup:diff` then `setup apply` against Personize using a repo secret. This never rebuilds or redeploys the engine service; the dispatcher reads guidelines and dispatch routes fresh from Personize on every cycle, so the change is live on the next cycle with no restart.

Dispatch routes are authored as manifest files under `manifests/core/dispatch-routes/*.json` — one route per file, same pattern as guidelines. See [docs/DISPATCH-ROUTES.md](DISPATCH-ROUTES.md) for the authoring guide and an important caveat on which `target_type` values actually work against the current operations catalog.

Set up the Action once: add `PERSONIZE_SECRET_KEY` (and `PERSONIZE_CRM_CONNECTION_ID` if applicable) as repo secrets under a `production` environment. After that, publishing content is just editing a file and pushing — no server visit, ever.
